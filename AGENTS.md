## 1. Что это за проект

Внутренняя система управления сервисом светодиодных экранов (LED-панелей) компании MS.
Три отдела работают с одним набором экранов:

- **Мониторинг** — замечает поломку панели, создаёт заявку
- **Контроль** — принимает заявку, распределяет в сервис, закрывает в архив
- **Сервис** — физически ремонтирует панели, возвращает на экран
- **ЗИП** — склад запасных панелей, проводов, хабов, ламелей

Плюс инфраструктурное:

- Уведомления в Telegram (сломаны в РФ — блокировка трафика)
- Парсинг писем тревог с китайского VNNOX через Gmail API
- Ежедневные таски (daily_checker) — напоминания по расписанию
- Планируется бот в МАХ (мессенджер VK) как замена/дубль TG

## 2. Роли агентов

Система проектирования строится по принципу **архитектор → кодеры**. Пути пересечения минимальны, точки синхронизации формализованы.

| Роль           | Агент                | Зона ответственности                                                    |
|----------------|----------------------|-------------------------------------------------------------------------|
| **Архитектор** | Claude Opus          | Архитектурные решения, постановка задач, ревью, конфликты, ADR          |
| **Backend**    | GPT-5 (кодер)        | Django/DRF, миграции, бизнес-логика, тесты, интеграции, инфра-скрипты  |
| **Frontend**   | Claude Design        | React SPA, Tailwind, UX, интеграция с API                               |
| **Human**      | Владелец проекта     | Финальное «да/нет», деплой, доступы к прод-БД и секретам               |

### Правила взаимодействия

1. **Кодер НЕ принимает архитектурные решения.**
   Если видишь, что задача ломает архитектуру или противоречит `ai-docs/00-architecture/` — **остановись, пиши в отчёт, отдавай архитектору**.
   Не «сделал как проще» — это причина для отклонения PR.

2. **Архитектор НЕ пишет прод-код.**
   Может написать референсный сниппет внутри задачи, но полноценная реализация — за кодером.

3. **Frontend и backend могут работать параллельно** — контракт API зафиксирован в `ai-docs/07-frontend/api-contract.md` **до** начала реализации фичи.
   Любое изменение контракта требует синхронизации через архитектора.

4. **Каждая выполненная задача = отдельный отчёт** в `ai-docs/08-reports/` по шаблону.

## 4. Стек (зафиксирован)

### Backend
- **Python 3.12**, **Django 5.1**, **DRF 3.15**
- **PostgreSQL 16**
- **Redis 7** — очереди (Streams) и кэш
- **SimpleJWT** — аутентификация SPA
- **django-environ** — конфиги
- **pytest + pytest-django** — тесты
- **ruff + black + mypy** — линтеры
- **structlog** — структурное логирование

### Frontend
- **React 18** + **TypeScript**
- **Vite** — сборка
- **TanStack Query** (React Query) — server state
- **Zustand** — клиентский стейт
- **React Router v6**
- **Tailwind CSS** + **shadcn/ui**
- **React Hook Form** + **zod** — формы и валидация

### Инфра
- **Docker + docker-compose** — локально и на проде
- **Gunicorn** + **Nginx** — прод
- **Redis** — брокер очередей
- **Systemd / supervisord** — процессы воркеров

**Celery мы не берём.** Остаёмся на Redis Streams с кастомным consumer-ом — причина: уже есть работающий `sender_tg_message.py`, Celery тут лишняя зависимость. Но **переезжаем с pubsub на Streams** (гарантированная доставка, повторы, DLQ).

## 5. Что строго запрещено

1. **НЕ коммитить** `.env`, `client_secret.json`, `token.pickle`, любые ключи. Если нашёл такое в репо — заводи critical-issue.
2. **НЕ удалять миграции.** Все миграции forward-only. Если нужна откатка — пишем новую миграцию.
3. **НЕ менять модели через `makemigrations --name auto`** — всегда осмысленное имя.
4. **НЕ импортировать между приложениями кроме как через публичный интерфейс** (см. `target-architecture.md`, раздел «слоистость»).
5. **НЕ писать бизнес-логику во views** — она в `services/` или `use_cases/`.
6. **НЕ использовать `from X import *`** — явные импорты.
7. **НЕ использовать `print()`** для логов — `logger = structlog.get_logger()`.
8. **НЕ писать `except Exception:` без конкретной причины** — ловим конкретные исключения.
9. **НЕ ломать FSM заявки** — все переходы состояний только через `ApplicationStateMachine.transition(...)`.
10. **НЕ делать `redirect(request.META['HTTP_REFERER'])`** — открытый редирект-уязвимость.
11. **НЕ сохранять текстовые файлы вне UTF-8 без BOM.** Для `.py/.ts/.tsx/.md` и других текстовых файлов действует `check_encoding.py` и правила из `.gitattributes`.

## 7. Контакты и эскалация

- Архитектурные вопросы и спорные решения — **архитектор** (Claude Opus в диалоге)
- Бизнес-вопросы (что именно должна делать кнопка, кто имеет доступ) — **владелец проекта**
- Конфликты между агентами — архитектор разрешает, решение фиксируется в `ai-docs/adr/` (Architecture Decision Record)

---

## 8. Версионность документации

`ai-docs/` — живой артефакт. Меняется вместе с кодом.

- Любое изменение архитектуры → PR должен содержать апдейт соответствующего документа.
- Добавил новую конвенцию → закрепил в `ai-docs/04-conventions/`.
- Принял сложное решение → создал `ai-docs/adr/ADR-XXX-<title>.md`.

**Кода без документации в этом проекте быть не должно.**

## 9. Если что-то пошло не так

Ты можешь:

- Перестать писать код и спросить — дешевле потерять 10 минут на вопрос, чем день на переделку.
- Заявить задачу как «неправильно сформулирована» — тогда архитектор её переписывает.
- Отказаться выполнять задачу, если считаешь её архитектурно вредной — это твоё право. В отчёте поясни причину.

**Последний пункт — единственное в этом документе, что ВАЖНЕЕ всего остального.**
Лучше отказ с аргументом, чем выполненная плохо задача.
