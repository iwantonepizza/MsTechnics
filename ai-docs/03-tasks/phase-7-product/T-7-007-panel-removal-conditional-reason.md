## Цель

Закрыть DV-S6 с уточнением владельца от 2026-05-17: снятие панели бывает в двух сценариях, и правила для `condition` у них разные.

---

## Контекст

### Сценарий A: снятие в рамках заявки

- Панель снимается сервисом при работе по заявке.
- Новое состояние панели должно быть выбрано явно.
- `condition` в форме обязателен.
- В `ActivityLog` должен фиксироваться `via_application_id`.

### Сценарий B: ручное снятие без заявки

- Панель снимается без закрытия заявки: ТО, демонтаж, перенос, замена по месту.
- Состояние панели по умолчанию сохраняется как есть.
- `condition` в форме optional / disabled.

---

## Что нужно было сделать

### Backend

- Доработать `PanelMover.remove_from_cell(...)`, чтобы он принимал optional `new_condition` и optional `application`.
- Добавить endpoint `POST /api/v1/panels/{id}/remove/` с полями:
  - `new_condition`
  - `comment`
  - `application_id`
- Провалидировать `new_condition` и корректность `application_id`.
- Записать `via_application_id` в `ActivityLog`, если снятие идёт из контекста заявки.

### Frontend

- Сделать modal снятия панели с двумя режимами:
  - из карточки заявки: обязательный выбор нового состояния;
  - ручное снятие: checkbox "Оставить как есть" и optional выбор нового состояния.
- Показать warning, если панель снимается вручную при наличии активной заявки.
- Добавить snapshot/smoke-тесты для обоих режимов.

- Не менять справочник `Condition` под этот кейс.
- Не запрещать ручное снятие панели только из-за активной заявки.
