# T-8-107. Prod UI stability: activity feed, cities retry, mobile camera

> **Статус:** in-progress
> **Исполнитель:** GPT-5 Codex
> **Дата старта:** 2026-06-04
> **Приоритет:** P0 prod hotfix
> **Scope:** Frontend + native production runtime + documentation

## Контекст

После prod-развёртывания владелец сообщил:

1. Нижняя лента последних действий бесконечно загружается.
2. На мобильном камера заменяется только ссылкой открытия.
3. Список городов/экранов иногда показывает ошибку загрузки.
4. В браузерной консоли повторяются `429 Too Many Requests` для `/cities/` и `/activity-log/`.

## Подтверждённые причины

- `since` для ActivityLog вычислялся при каждом render. Значение с новыми миллисекундами входило в
  TanStack Query key, поэтому каждый render создавал новый запрос и следующий render.
- Общий query retry повторял `429` ещё два раза и усиливал запросный шторм.
- Кнопка повтора списка экранов вызывала только `refetch()` экранов, даже если упал запрос городов.
- `DisplayCameraCard` через пять секунд удалял iframe из DOM и заменял его fallback-блоком. На
  медленной мобильной сети рабочий поток не получал времени догрузиться.
- Prod gunicorn использовал sync workers для долгоживущего SSE. Три workers регулярно получали
  `CRITICAL WORKER TIMEOUT`, перезапускались и дополнительно ухудшали доступность API.
- Prod nginx/access и gunicorn error logs записывали SSE query-token вместе с URL.

## План

1. Сделать `since` стабильным между render и менять только при переключении периода.
2. Не повторять `429` автоматически; добавить явные состояния ошибки и ручной retry.
3. Повторять загрузку и городов, и экранов одной кнопкой.
4. Не удалять iframe камеры по таймауту; всегда сохранять внешний путь открытия.
5. Вернуть фильтр ленты по виду действия на главной.
6. Закрепить gevent workers и отдельный nginx SSE location без buffering/access log.
7. Прогнать frontend/backend проверки, развернуть через git и проверить prod-логи.

## Критерии приёмки

- [ ] Лента делает один запрос на выбранный период/тип, не создаёт render-loop.
- [ ] `429` не повторяется автоматически.
- [ ] Ошибка ленты отображается явно и допускает ручной повтор.
- [ ] Повтор списка экранов повторяет запросы `/cities/` и `/displays/`.
- [ ] Камерный iframe остаётся в DOM после таймаута и на мобильном layout.
- [ ] Лента главной фильтруется по 1/2 месяцам и виду действия.
- [ ] SSE обслуживается gevent workers без регулярных gunicorn timeout.
- [ ] SSE query-token не пишется nginx в access log.
- [ ] Targeted и full проверки зелёные.
- [ ] Hotfix развёрнут и проверен на prod.

## Ограничения

- Если поставщик камеры запрещает iframe через `X-Frame-Options`, CSP или mixed content, frontend
  не может снять ограничение браузера. Внешняя ссылка должна оставаться доступной.
- Глобальный backend throttle не повышается без необходимости: сначала устраняется клиентский
  request-loop, который нарушает нормальный профиль нагрузки.
- Старые логи с уже истёкшими JWT не удаляются автоматически: очистка/retention логов требует
  отдельного операционного решения.
