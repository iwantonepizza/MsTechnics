# T-8-111. Owner prod UX, media reconciliation and history reset

> **Статус:** in-progress
> **Исполнитель:** GPT-5 Codex
> **Дата старта:** 2026-06-04
> **Приоритет:** P0/P1 production stabilization
> **Scope:** Frontend + API + management commands + production data + documentation

## Контекст

После T-8-107 владелец запросил второй пакет production-исправлений:

1. Сделать камерный блок без полос прокрутки и добавить переиспользуемый минималистичный
   загрузчик с волной точек и состояниями «Загрузка» / «Суперсимметрия».
2. Перенести VNNOX из отдельной вкладки правой панели в колонку подробностей экрана.
3. Показывать истории и заявки за всё время с постепенной подгрузкой при прокрутке.
4. Удалить неработающие быстрые ссылки «История» / «Заявки» у строк экранов и показывать
   количество заявок.
5. Исправить загрузку фотобанка, просмотр/загрузку схемы и проекта, не показывать ссылки на
   отсутствующие media-файлы.
6. Сверить production media, очистить отсутствующие ссылки и подтверждённые заглушки.
7. После обязательного свежего backup очистить операционные журналы по явному запросу владельца.

## Граница очистки данных

Очистка затрагивает только журнальные/исторические записи:

- `ActivityLog`;
- `ApplicationEvent` и `ApplicationHistoryReport`;
- `DepartureHistoryReport`;
- `PanelHistoryReport`;
- `DisplayHistoryReport`;
- `DailyTaskHistoryReport`.

Не удаляются пользователи, заявки, экраны, панели, ячейки, справочники, текущие состояния,
уведомления и VNNOX alarm events. Учётная запись `thecreator` и остальные учётные записи должны
остаться неизменными. Команда по умолчанию работает в dry-run и требует явный флаг применения.

## План

1. Провести read-only аудит production: журналы, заявки, media, 404/429 и runtime-логи.
2. Реализовать переиспользуемый loader и исправить камерный блок.
3. Перестроить display layout и лениво загружать VNNOX в колонке подробностей.
4. Перевести ActivityLog/заявки на cursor-based infinite scroll без ограничения 1/2 месяца.
5. Исправить действия строки экрана и добавить количество заявок.
6. Добавить API/UI загрузки media, корректное поведение отсутствующих файлов и команду сверки.
7. Добавить безопасную команду очистки истории с тестами.
8. Прогнать проверки, оформить отчёт, сделать свежий backup, deploy через git и production-аудит.

## Критерии приёмки

- [ ] Камера на desktop не показывает полосы прокрутки; загрузчик доступен как общий компонент.
- [ ] VNNOX расположен в колонке подробностей и запрашивается только после открытия блока.
- [ ] Истории и заявки доступны за всё время и догружаются по мере прокрутки.
- [ ] У строки экрана нет неработающих ссылок; при наличии заявок показан их счётчик.
- [ ] Фотобанк загружает файлы; схема и проект открываются крупно и допускают загрузку.
- [ ] API не выдаёт URL отсутствующего media; prod-ссылки и подтверждённые заглушки сверены.
- [ ] Очистка истории выполняется только после backup и не меняет пользователей/текущие сущности.
- [ ] Targeted/full проверки зелёные; код развёрнут через git и production проверен.

## Ограничения

- Внешний camera iframe может быть ограничен самим поставщиком. Ссылка открытия остаётся доступной.
- Файлы, которые нельзя однозначно признать заглушками, автоматически не удаляются.
- Runtime media и production dump не коммитятся в git.
