# Git Workflow

## Ветки

- `main` — прод. Protected. Пишется только через PR. CI обязателен.
- `develop` — не используем (trunk-based).
- `feature/<task-id>-<kebab-name>` — фича-ветки. Например `feature/T-1-007-env-secrets`.
- `hotfix/<short-desc>` — срочные фиксы прода минуя roadmap. Например `hotfix/tg-bot-healthcheck`.

## Правила

1. **Ветка = одна задача.** Не склеивай 3 задачи в один PR «раз уж всё равно в этом файле».
2. **PR должен закрывать задачу полностью** или явно отмечать что остаётся. Полу-сделанных PR не должно быть.
3. **Живёт ветка — максимум неделю.** Долгая ветка = конфликты и страх мёржа.
4. **Rebase на `main` перед мёржем** — линейная история.

## Протокол

```
# 1. Свежая main
git checkout main && git pull

# 2. Новая ветка от main
git checkout -b feature/T-2-005-application-event-model

# 3. Коммиты (см. commit-format.md)
# ...

# 4. Перед открытием PR — синкнуться
git fetch origin
git rebase origin/main
# фиксим конфликты
git push --force-with-lease

# 5. Открываем PR в main
# - описание по шаблону (см. ниже)
# - ссылка на задачу
# - ссылка на отчёт в ai-docs/08-reports/

# 6. Ревью → правки → мёрж (squash or rebase, без merge-commit'ов)
```

## PR Template

```markdown
## Что
<кратко, одна-две строки>

## Зачем
<ссылка на задачу в ai-docs/03-tasks/>

## Как
<ключевые решения, если что-то нетривиально>

## Отчёт
<ссылка на ai-docs/08-reports/T-X-XXX.md>

## Чеклист
- [ ] Задача отмечена в трекере (status: review)
- [ ] Tests pass locally (`pytest`)
- [ ] Linters pass (`ruff`, `black`, `mypy`)
- [ ] Coverage >= 80% для нового кода
- [ ] Документация обновлена (если менял архитектуру)
- [ ] Миграции проверены на dev-копии прод-БД (если есть миграции)
- [ ] Нет секретов в коде
- [ ] Нет `print()` / `console.log()` / debug-кода
- [ ] Отчёт в ai-docs/08-reports/ заполнен

## Связанные PR
<если есть зависимости>
```

## Ревью

- Ревьюер: **архитектор** (Claude Opus в диалоге)
- SLA ревью: < 24ч для standard, < 2ч для hotfix
- Количество итераций не ограничено, но если больше 5 — стоит перепроектировать задачу

## Как отвечать на комментарии

- Зелёная галочка ✅ — принято и исправлено
- «Не согласен потому что Х» — это ОК, обсуждаем
- «Исправил бы, но это выходит за рамки задачи» — ОК, заводи новую задачу в tracker

## Rebasing vs merging

- Внутри feature-ветки — rebase на main каждый день
- Мёрж в main — **squash-merge или fast-forward**. Merge-commit'ов избегаем.
- Исключение: hotfix-ы могут идти обычным merge, чтобы оставить явную ветку

## После мёржа

```
# удалить локальную ветку
git branch -d feature/T-X-XXX

# удалить remote (если платформа не сделала)
git push origin --delete feature/T-X-XXX
```

## Что НЕ делать

- **НЕ force-push в main.** Никогда.
- **НЕ коммит `--no-verify` без причины.** Если pre-commit падает — фикси, не обходи.
- **НЕ коммить напрямую в main.** PR или не коммит.
- **НЕ ломай CI ради «быстро, потом починю».** CI зелёный → PR, не иначе.
- **НЕ меняй чужие commits в merged-ветках.** История — артефакт.

## Что делать с секретом в истории

Если случайно закоммитил `.env`:
1. **НЕ думай** что `git rm` + новый коммит помогут — секрет остался в истории
2. Отзови секрет немедленно (rotate password, revoke token)
3. `git filter-repo --invert-paths --path <file>` или BFG-repo-cleaner
4. `git push --force` (в экстренном режиме, со всеми)
5. Оповести команду
6. Напиши post-mortem: `ai-docs/08-reports/incident-YYYY-MM-DD-<описание>.md`
