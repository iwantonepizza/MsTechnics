Эти брифы помогают дизайнеру понять конкретику, но **не диктуют JSX**.

### Экран 1. Display View / Мониторинг

**URL:** `/monitoring/:city/:display`

**API:** `GET /api/v1/displays/{slug}/` — тот же что в Сервисе, только role в JWT другая.

**Что одинаковое с Display View / Сервис:**
- Хедер
- Title-bar (название экрана + город + UTC + метрика 98/100)
- Сетка ячеек 10×10 с линейками A-J / 01-10 + легенда
- KV-блок выбранной панели справа
- TimeLine «История места / История панели»
- Footer с SSE-индикатором + горячими клавишами

**Что другое:**

| Элемент | Сервис | Мониторинг |
|---|---|---|
| Кнопка «Снять панель» | есть | **нет** |
| Кнопка «Поставить панель» | есть | **нет** |
| Inline-смена состояния | full dropdown | **только** `work → problem`, остальное disabled |
| Кнопка «🆕 Создать заявку» | нет | **есть, активна на красных ячейках** |
| Вкладки заявок (нижняя секция) | 7: Запросы / В работе / Выполненные / Архив / История / Невозможные / Все | **5: Мои создания / В работе / Архив / История / Все**. У монитора нет «Запросов» как у контролёра — он не разбирает их |
| Кнопки на карточках заявок | принять / в сервис / выполнено / unable | **только просмотр + кнопка-глазик "история заявки"** |

**Ключевое для UX:**
- Когда монитор кликает на красную ячейку — справа сразу должна быть видна кнопка «🆕 Создать заявку» крупно, потому что это его **основное действие**
- Если у панели уже есть активная заявка — кнопка disabled с tooltip «Заявка уже создана: ID-XXXX»
- У монитора нет роли `service`, поэтому панели в сервисе он видит как read-only (фильтр уже на API)

### Экран 2. Display View / Контроль

**URL:** `/control/:city/:display`

**Что одинаковое с Сервисом:** хедер, сетка, panel-info, timeline, footer

**Что другое:**

| Элемент | Сервис | Контроль |
|---|---|---|
| Вкладки заявок | Запросы / В работе / Выполненные / Архив / История / Невозможные / Все | **Запросы / Принятые / Отправленные в сервис / Выполненные сервисом / Архив / Невозможные / Все** |
| Кнопка действия на карточке | взять в работу [R] / выполнено [D] | **Принять [A] / В сервис [S] / Архивировать [V]** |
| Inline-смена состояния | да | да, но **через подтверждение** (не сразу) |
| Кнопка «Создать заявку» | нет | **есть** (контролёр может) |

**Ключевое:**
- Вкладка «Запросы» — главная, на неё контролёр приходит первым делом
- В каждой строке-заявке должно быть видно: ID, экран, ячейка, кто создал (монитор), когда, комментарий с 3-line clamp
- Назначение исполнителя — встроено в действие «В сервис»: модалка с dropdown executor + поле комментария
- Действие «Архивировать» — модалка confirm с предупреждением «Заявка уйдёт в архив, не вернётся»

│  │ Ленина 60        🟢                │  →   │  только;     │
│  │ 8×6 · 0 проблем · открыть          │      │  у service   │

**Ссылки на каждом экране-карточке:**

- Открыть → переход на Display View текущего отдела

### Экран 4. ZIP Overview

**URL:** `/zip` (общий) и `/zip/:display` (фильтр по экрану)

**API:**
- `GET /api/v1/panels/?department=zip|hand|service|monitor`
- `GET /api/v1/storage/wires?display=<slug>`
- `GET /api/v1/storage/hubs?display=<slug>`
- `GET /api/v1/storage/lamels?display=<slug>`

**Layout:**
```
┌─────────────────────────────────────────────────────────────────┐
│ Header                                                          │
├─────────────────────────────────────────────────────────────────┤
│ Title: ЗИП — все экраны | [Фильтр по экрану ▼]   [+ Панель]    │
├──────────┬──────────┬──────────┬──────────┬───────────┬─────────┤
│ ЗИП      │ На руках │ Сервис   │ В работе │ Расходники│ История │
│ (12)     │ (3)      │ (5)      │ (84)     │           │         │
├──────────┼──────────┼──────────┼──────────┼───────────┼─────────┤
│ P-00012  │ P-00045  │ P-00078  │ P-00001  │ Ламели    │ ▷ 14:30 │
│ работает │ моргает  │ ремонт   │ работает │ Q200: 47  │ замена  │
│          │          │          │          │ Q300: 12  │ P-00012 │
│ P-00013  │ P-00046  │          │ P-00002  │           │ → ЗИП   │
│ работает │ работает │          │ работает │ Хабы      │         │
│          │          │          │          │ A-100: 5  │ ▷ 13:15 │
│ ...      │ ...      │ ...      │ ...      │ B-200: 2  │ ...     │
└──────────┴──────────┴──────────┴──────────┴───────────┴─────────┘
```

**Особенности:**
- 4 колонки панелей по `department` (count в заголовке)
- Каждая плитка панели — `<PanelBadge>` с цветом из condition + ID + краткое состояние
- Drag-and-drop (опционально, не блокер) — между колонками
- Колонка «Расходники» — wires/hubs/lamels с `count` (число шт.) + кнопка `+/-` для admin/control
- Колонка «История» — последние 20 записей `ActivityLog` с кликабельностью на target
- На `/zip/:display` — все колонки автоматически фильтруются по этому экрану + в шапке breadcrumb «← все экраны / Колизей»

**Заметки по UX:**
- Список панелей в каждой колонке может быть длинным (84+ в монитор-колонке) — virtual scroll
- Click на панель → правая панель (`<PanelInfoCard>`) выезжает с действиями (как на Display View)
- При попытке перетащить панель из «В работе» в «ЗИП» — если есть активная заявка, показывается toast «Сначала закройте заявку»

### Экран 5. Модалки transition

**Не отдельный экран** — модалки поверх Display View.

**Список:**

| Action | Where | Form |
|---|---|---|
| Принять заявку (control) | Display View / Контроль | comment? |
| В сервис | Display View / Контроль | comment + executor (dropdown) |
| Взять в работу | Display View / Сервис | comment? |
| Выполнено | Display View / Сервис | comment + photo (опционально) |
| Невозможно | Display View / Сервис | **comment обязателен** + photo |
| Архивировать | Display View / Контроль (после done/unable) | comment? + warning |
| Удалить заявку | Display View / Мониторинг | confirm + 5-min window timer |
| Создать заявку | Display View / Мониторинг + Контроль | comment + photo (опционально) |
| Сменить состояние панели | Display View любой | dropdown условий + comment |
| Сменить отдел панели | ZIP / Display View | dropdown department + comment + warning если есть active app |
| Поставить панель в ячейку | Display View / Сервис | dropdown свободных панелей + comment |
| Снять панель с ячейки | Display View / Сервис | dropdown условия (work/problem/broken) + comment |

**Универсальный шаблон модалки:**
```
┌────────────────────────────────────────────┐
│ Заголовок действия            [×]           │
├────────────────────────────────────────────┤
│ Контекст: ID-4567 · Колизей · A-3          │
│ Текущий статус: [pill]                      │
│ → Новый статус: [pill]                      │
├────────────────────────────────────────────┤
│ Комментарий *                               │
│ [textarea ...                            ]  │
│                                             │
│ Фото (опционально)                          │
│ [□ загрузить...]                            │
├────────────────────────────────────────────┤
│                  [Отмена]  [Подтвердить S]  │
└────────────────────────────────────────────┘
```

**Шорткаты:** Esc = Отмена, Ctrl+Enter / Cmd+Enter = Подтвердить.

**Optimistic state:**
- Submit → кнопка disabled + spinner
- Server OK → закрытие, мгновенно обновляется UI через TanStack `setQueryData`
- Server error → toast + кнопка снова active

## Часть 4. Архитектурное замечание для дизайнера

В предыдущем раунде я заметил, что ты захардкодил hex-коды в JSX. Ты исправил через CSS-переменные — отлично. **Применяй то же правило ко всем новым экранам**: цвета только через `--token-name`, статусы через `var(--ok)/--warn)/...`, никаких `#fff` и `#000` в JSX.

Также — ты добавил скелетоны в tokens.css v2. **Используй их** на всех новых экранах. Каждый async-блок (список, детальная инфа) должен иметь skeleton-state.

## Часть 6. Когда возвращать ответ

- На каждый экран сначала **эскиз** (текст/ASCII), потом — JSX.
- Эскиз → ревью → правки → JSX → ревью → правки → принято.
- 2-3 итерации на экран — норма. Меньше — слишком быстро (пропустили детали). Больше — обсудим, может что-то системно не так.
