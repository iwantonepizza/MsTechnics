# Design Brief — MsTechnics SPA

> Этот документ — **главная точка входа** для Claude Design. Остальные файлы
> в `07-frontend/` — детализация по отдельным аспектам.

---

## 1. Что это и зачем

Внутренний инструмент для трёх отделов (Мониторинг, Контроль, Сервис) + склад (ЗИП) для обслуживания LED-экранов. Пользователи работают **смены по 8-12 часов**, смотрят в интерфейс весь день, на разных девайсах: от офисного 27" монитора до 13" ноутбука и 10" планшета сервисника в поле.

**Это не клиентский продукт.** Здесь важны плотность информации, скорость переходов, предсказуемость, а не «wow-эффект». Интерфейс должен быть похож на Linear / Superhuman / Jira (плотный, клавиатурный), а не на Airbnb / Stripe (просторный, маркетинговый).

### Что он заменяет
Существующий Django-templates + ванильный JS интерфейс. Скриншоты исходника — в `reference/legacy-screenshots/` (сделаем по запросу). Шаблоны на чтение: `templates/`, `*/templates/`.

### Что изменяется концептуально
- Был: перезагрузка страницы на каждое действие, модалки через fetch HTML в overlay.
- Стал: SPA с роутингом, optimistic updates, real-time через SSE, модалки — реактивные.

## 2. Кто пользователь

### 2.1. Персоны

**Мониторщик (Катя)**
- Смотрит в общий мониторинг-экран 8 часов, замечает проблемы с панелями
- Действие: отметить панель как «не работает» → создать заявку
- Девайс: офисный десктоп, 1920×1080, FullHD часто двухмониторная
- Нужна максимальная плотность (много экранов в городе)

**Контролёр (Миша)**
- Принимает заявки от мониторинга, решает что делать: в сервис, архив, невозможно
- Комментирует, назначает исполнителя, трекает статусы
- Девайс: десктоп, но иногда ноутбук
- Нужен быстрый просмотр заявки + быстрые действия

**Сервисник (Артём)**
- В поле с планшетом или на пункте ремонта
- Смотрит заявку, делает ремонт, закрывает с комментарием + фото
- Девайс: планшет 10", иногда десктоп
- Нужен крупный тач-френдли UI, минимум полей

**Кладовщик ЗИП (Валера)**
- Ведёт учёт свободных панелей, проводов, хабов
- Переводит панели между складом/сервисом/экраном
- Девайс: десктоп
- Нужна таблица с фильтрами + история перемещений

**Админ (владелец)**
- Настраивает состояния, цвета, иконки, пользователей, доступы
- Редкие действия, но критичные

### 2.2. Что их объединяет
- **Разрешения строгие.** У Кати НЕТ доступа в сервис. У Артёма НЕТ доступа в контроль. Навигация должна показывать только то, что можно.
- **Аудит важен.** Всё логируется: кто что сделал, когда, по какой заявке.
- **Русский язык,** клавиатура — кириллица-латиница.
- **Часовой пояс — МСК по умолчанию,** но у городов (Ижевск, Казань, и т.д.) может отличаться.

## 3. Глобальные принципы UI

### 3.1. Визуальный стиль

Темный интерфейс. Плотный. Информационный.

**Цветовая логика:**
- Нейтральный фон (`zinc-900`, `zinc-800`)
- Акцентный цвет — один (например, `amber-400` — «рабочий» цвет MS, есть в лого)
- Семантика через цвет: заявка/состояние — из БД (см. модель `Color`), не захардкодим
- Состояния (hover, active, disabled, chosen) — чёткие, контрастные

**Типографика:**
- Inter / IBM Plex Sans / системный стек
- Моноширинный для ID: JetBrains Mono / ui-monospace
- Размеры: 12px / 14px / 16px / 20px / 24px. Больше — крайне редко.
- Высота строк плотная, 1.3-1.4.

**Плотность:**
- Отступы между элементами 8-16px, не 24-48. Это не лендинг.
- Карточка заявки должна помещать 8 штук в колонке на 1920×1080 без скролла.

**Форма:**
- Скругления 6-8px максимум. Не мягкие капсулы.
- Границы 1px, тонкие.
- Тени — только на модалках/поповерах, не везде.

### 3.2. Поведение

- **Optimistic updates везде.** Нажал «принять заявку» → статус меняется мгновенно, запрос идёт на сервер. Если упало — откат + toast.
- **Без «покажу спиннер на 2 секунды».** Любая загрузка > 500мс → skeleton. Меньше — ничего не показываем, успеет прийти.
- **Keyboard-first.** Везде, где можно — шорткаты. Список горячих клавиш — в `states-and-flows.md`.
- **Hover-tooltip'ы есть,** но не перегружены. Только там где нужен контекст.
- **Never refresh the page.** Все действия в SPA.
- **URL — source of truth.** `/service/izhevsk/display-1?cell=03&app_box=at_work` → при перезагрузке состояние восстанавливается.

### 3.3. Доступность
- Контраст текста AAA для основного, AA для вторичного
- Все интерактивные элементы — кнопки, не div'ы с onClick
- Фокус-стейт обязателен
- aria-labels для иконок без текста

### 3.4. Адаптивность
- Desktop 1920+ — основной сценарий, должна быть максимальная плотность
- Laptop 1366 — поддерживается, один-два элемента могут сжаться
- Tablet 1024 — сервисный сценарий, тач-френдли, некоторые дашборды могут быть недоступны
- Mobile < 768 — НЕ поддерживается. Редирект на «откройте с планшета/компьютера».

## 4. Информационная архитектура

```
/ (login если не авторизован)
/menu                             — главное меню (дашборд всех отделов, в которые у юзера доступ)
/monitoring                       — список городов → экранов (доступ: monitoring)
/monitoring/:city/:display        — рабочее окно монитора экрана
/control                          — список городов → экранов (доступ: control)
/control/:city/:display
/service                          — список городов → экранов (доступ: service)
/service/:city/:display
/zip                              — общий вид склада
/zip/:display                     — склад с фильтром по экрану
/lk                               — личный кабинет (профиль, смена пароля)
/admin                            → Django admin (ссылка)
```

Навигация в шапке: «Главная / Мониторинг / Контроль / Сервис / ЗИП / Выход». Видны только те, в которые у юзера есть разрешение.

## 6. Ключевые компоненты

Полный список — в `components.md`. Здесь — те, что определяют стиль проекта.

### 6.1. `<Grid>` — сетка ячеек экрана
- 10×10 или другой размер, определяется экраном
- Каждая ячейка: цветная (из `application.status.color`), с номером ячейки, иконкой состояния
- Hover: выделение, tooltip с ID панели
- Click: выбор ячейки, URL обновляется

### 6.2. `<ApplicationCard>`
- ID, панель, экран, слот, статус, исполнитель, время
- Цвет ID = `application.status.color.hex`, текст — `color_text.hex`
- Комментарий — обрезается в 3 строки (см. задачу владельца #2). Полный — в hover-popover.
- Кнопки действий — в зависимости от статуса заявки и роли

### 6.3. `<StatusPill>` — индикатор состояния
- Маленький бейдж с цветом + иконкой
- Используется везде: в списках, в ячейках, в деталях

### 6.4. `<CommentsHistory>` — лента комментариев к панели/заявке/ячейке
- По датам, с пользователем и временем, hover — подробности
- Кликабельная (в задаче #11: открыть заявку из лога)

### 6.5. `<DisplayForm>` — форма с фиксированным набором полей
- Поля приходят из бэка (`DisplaySpec`), рендер по типам
- Хотя и статический, но форма изолирована в отдельный компонент

### 6.6. `<Modal>` — диалоги действий
- `confirm_change_department`, `panel_condition_confirm`, `create_application`, `add_service_comment` и т.д.
- Все модалки — сущности, а не HTML-фрагменты с сервера
- Submit → мутация → optimistic update → закрытие

### 6.7. `<Tabs>` — вкладки заявок (Запросы / В работе / Выполненные / Архив / История / Невозможные / Все)
- Сохраняют сортировку независимо (задача владельца #6)

## 7. Ключевые пользовательские потоки

Детально — в `states-and-flows.md`. Здесь — самое главное.

### Поток 1: мониторщик создаёт заявку
1. Видит красную ячейку на экране
2. Кликает → на правой панели появляются детали
3. Нажимает «Не работает» → модалка выбора состояния (сломана/моргает/и т.п.)
4. Подтверждает → статус панели меняется, автоматически создаётся заявка со статусом `sent_to_control`
5. Контролёр получает уведомление (MAX/TG)

### Поток 2: контролёр принимает заявку
1. Открывает `/control/:city/:display`
2. Во вкладке «Запросы» видит новую заявку
3. Кликает «Принять» → заявка в `apply_in_control`
4. Добавляет комментарий и исполнителя
5. «Отправить в сервис» → `sent_to_service`, уведомление сервиснику

### Поток 3: сервисник закрывает заявку
1. `/service/:city/:display`, вкладка «Запросы»
2. Берёт заявку в работу → `work_in_service`
3. Физически заменил панель → «Выполнено» / «Невозможно»
4. Архивирует
5. Лог закрывается в `ActivityLog`

## 8. Real-time

- SSE (не WebSocket) от бэка на канал `user:<id>:notifications`
- Сообщения: `application.created`, `application.changed`, `panel.condition_changed`, `departure.created`
- На клиенте: invalidate соответствующих TanStack Query кешей → UI перерисовывается
- Если SSE отвалилось — fallback на `refetchInterval: 60000` (раз в минуту)

---

## 9. Работа с ошибками

- 401 → refresh JWT, если упал — редирект на /login
- 403 → toast «Нет доступа» + логирование
- 422 (валидация) → подсветка полей в форме
- 5xx → toast «Упс, сервер. Мы уже знаем» + Sentry
- Offline → баннер «Нет подключения. Изменения не сохраняются»

## 11. Что **не** надо делать

1. **Не переизобретай доменную модель.** Заявка, панель, ячейка, экран — всё как в `ai-docs/00-architecture/domain-model.md`. Если кажется что надо иначе — вопрос архитектору.
2. **Не тащи Redux / MobX.** Zustand + TanStack Query достаточно.
3. **Не пиши свой UI-kit.** Берём shadcn/ui (copy-paste) + Tailwind. Кастомизируем.
4. **Не используй анимации «для красоты».** Только там, где помогает понять изменение состояния.
5. **Не храни токены в localStorage.** Access — в памяти, refresh — в httpOnly cookie.
6. **Не делай бесконечную прокрутку для заявок.** Пагинация + фильтры. Люди работают по заявкам, а не листают.
7. **Не используй эмодзи в коде компонентов.** Иконки — Lucide. Эмодзи только там, где это **данные** (иконка состояния панели из БД).
