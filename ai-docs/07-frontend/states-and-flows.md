# States and Flows

Описание состояний страниц и ключевых пользовательских переходов.

---

## 1. Стейт-машина заявки (клиент отражает бэк)

```
[null] ──создание──► sent_to_control
                            │
                            │ принять
                            ▼
                       apply_in_control
                            │
                            │ отправить в сервис
                            ▼
                       sent_to_service
                            │
                            │ взять в работу
                            ▼
                       work_in_service
                            │
                       ┌────┴────┐
                       │         │
              выполнено          невозможно
                       │         │
                       ▼         ▼
                     done       unable
                       │         │
                       │         │ архив
                       └────┬────┘
                            ▼
                     archive_done | archive_unable
```

Кнопки отображаются только для доступных переходов, с учётом роли пользователя. Запрещённые переходы — не показываются, не серым (визуальный шум).

## 2. Потоки, детально

### Поток A: мониторщик создаёт заявку

1. Юзер открывает `/monitoring/izhevsk/central`
2. `DisplayGrid` рендерит 100 ячеек, красные = с проблемной панелью
3. Клик на красную ячейку → URL `?cell=03`, правая панель показывает панель
4. Клик «Не работает» (кнопка `<ConditionChangeButton>`)
5. Модалка `<PanelConditionConfirmModal>` — выбор состояния из dropdown (moргает, сломана, разбита) + комментарий
6. `onConfirm`:
   - Optimistic: панель меняет цвет, модалка закрывается
   - POST `/api/v1/panels/<id>/` с `{ condition_id }`
   - Затем POST `/api/v1/applications` с `{ panel_id, cell_id, comment }`
   - Toast «Заявка создана»
7. SSE: бэк шлёт `application.created` в канал контроля → у контролёров обновляется список

**Если ошибка:**
- Откат optimistic state
- Toast ошибки с кодом/сообщением

### Поток B: контролёр принимает → отправляет в сервис

1. `/control/izhevsk/central`, вкладка «Запросы»
2. Клик на заявку → open `<ApplicationCard variant="full">` в правой панели или инлайн-раскрытие
3. Кнопка «✅ Принять» → POST `/transition { target: apply_in_control }`
4. Кнопка «📤 В сервис» → модалка с выбором исполнителя + комментарий → POST `/transition { target: sent_to_service }`
5. SSE: у сервисников появляется заявка

### Поток C: сервисник делает ремонт

1. `/service/izhevsk/central`, вкладка «Запросы»
2. Видит новую заявку со статусом `sent_to_service` (цвет оранжевый)
3. Клик «🔧 В работу» → заявка в `work_in_service`
4. Физически меняет панель — это офлайн-действие
5. Вернулся → вкладка «В работе» → «✅ Выполнено»
6. Модалка: комментарий + фото
7. POST `/transition { target: done }` с multipart
8. Если снял панель → отдельное действие `<MovePanelFeature remove-from-cell>`

## 3. Состояния страниц (по экрану)

### DisplayViewPage state

```
{
  display: Display | null,
  selectedCell: string | null,      // из URL
  selectedPanel: string | null,     // вычисляется из selectedCell или из URL
  activeTab: 'received' | 'at_work' | ...,
  openApplicationId: number | null, // из URL ?app_id=
  tabSortings: { [tab]: SortOption }, // локально, не в URL
  modalStack: Modal[],               // какие модалки открыты
}
```

### Переходы URL
- `?cell=03` добавляется/меняется при клике на ячейку
- `?panel_id=P-12345` — альтернатива cell, если панель выбрана не через ячейку
- `?app_box=at_work` — вкладка
- `?app_id=4567` — открытая заявка (модалка или expanded-state)

## 4. Загрузочные состояния

### Правило: два порога
- < 300мс → ничего не показываем (успеет прийти, мигание раздражает)
- 300мс — 2с → Skeleton
- > 2с → Skeleton + текст «Всё ещё загружаю...»
- > 10с → Error state с retry («Сервер не отвечает. Попробовать ещё раз»)

### Skeleton-ы нужны для:
- DisplayGrid (пока грузится экран)
- ApplicationsList (при смене вкладки)
- PanelInfoPanel (при смене ячейки)
- DashboardBlock

---

## 5. Пустые состояния (empty)

Не «Ничего нет», а:
- «Заявок во вкладке "Запросы" нет. Всё под контролем.»
- «У тебя нет доступа к городам. Обратись к администратору.»
- «Панели на складе отсутствуют. Обратись в сервис или смени фильтр.»

Всегда с контекстом и возможным действием.

## 6. Горячие клавиши

| Комбинация | Действие |
|---|---|
| `g m` | На главное меню |
| `g s` | В сервис |
| `g c` | В контроль |
| `g z` | В ЗИП |
| `Esc` | Закрыть модалку / снять выбор ячейки |
| `/` | Фокус на поиск (если есть) |
| `?` | Открыть shortcut-help |
| `j / k` | Вниз / вверх по списку заявок |
| `Enter` | Открыть выбранный элемент |
| `a` | Принять заявку (на выбранной, в контроле) |
| `r` | Отправить в работу (в сервисе) |
| `d` | Выполнить (на выбранной в работе) |

Реализовать через `react-hotkeys-hook` или свой hook. Не через глобальный keydown-listener в корне.

## 7. Поведение после action

Всегда три этапа:
1. **Optimistic update** — UI меняется мгновенно
2. **Network** — запрос идёт на сервер
3. **Reconcile** — при успехе / ошибке — актуализировать из ответа или откатить

Примеры:
- Принять заявку: локально меняем статус → запрос → если 200, invalidate query → если 4xx/5xx, revert + toast
- Создать заявку: временный ID `tmp-xxx`, подставляем в список → на успехе заменяем на реальный → на ошибке убираем + toast

---

## 8. Real-time reconciliation

- SSE-сообщение `application.transition` → клиент находит в кеше эту заявку
- Если есть → `queryClient.setQueryData` сразу с новыми полями
- В любом случае `queryClient.invalidateQueries(['applications', filter])` чтобы гарантировать консистентность

При этом:
- Не перерисовываем, если тот же пользователь сам вызвал transition (anti-flicker)
- Показываем toast «Статус заявки ID-4567 изменён пользователем Misha» если это не мы

---

## 9. Обработка конфликтов

Optimistic concurrency через `If-Match: <etag>`. Бэк возвращает 409 если версия не совпала. Клиент:
- Показывает модалку «Заявка была изменена другим пользователем»
- Загружает актуальное состояние
- Предлагает повторить или отменить
