# Reports — отчёты по задачам

Каждая выполненная задача оставляет отчёт в этой папке. Отчёт живёт в git вместе с кодом — это источник правды о том, что было сделано.

## Зачем отчёты

1. **Архитектору** — понять, что кодер реально сделал, прежде чем идти в PR
2. **Будущему разработчику** — понять, почему то или иное место такое. Git blame → PR → отчёт → знаешь зачем.
3. **Самому кодеру** — проверить себя через месяц: «что я сделал и правильно ли»

## Что НЕ отчёт

- Не дневник
- Не переписывание PR-описания
- Не описание архитектуры (это в `00-architecture/`)

## Формат

Используй `TEMPLATE.md` как основу. Имя файла = `<task-id>.md` (например `T-1-001.md`). Для инцидентов — `incident-YYYY-MM-DD-<краткое-описание>.md`.

## Длина

1-2 экрана текста. Больше = отчёт читать не будут.

## Жизненный цикл

1. Кодер берёт задачу → создаёт файл-заготовку с одним `## План` разделом
2. В процессе работы заметки ведёт **здесь же**, не в черновике
3. При открытии PR — заполняет все разделы
4. Ревьюер читает отчёт **перед** чтением diff'а
5. После мёржа файл остаётся как артефакт

## Инциденты

Для инцидентов (прод упал, утечка секрета, и т.п.) — формат тот же, но:
- Приоритет: P0
- После написания — обязательно разбор с владельцем
- Выводы добавляются в `01-current-state/security-issues.md` или `performance-issues.md`
