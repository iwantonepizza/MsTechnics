## Что сделано

- Добавлен `MAX_WEBHOOK_SECRET` в settings и `.env.example`.
- `MaxChannel` теперь добавляет inline-кнопки из `context["actions"]` в `reply_markup.inline_keyboard`.
- Добавлен endpoint `POST /api/v1/integrations/max/webhook`.
- Webhook валидирует `X-MAX-Secret` / `X-Max-Webhook-Secret`, если `MAX_WEBHOOK_SECRET` задан.
- Обработаны входящие `message_created`:
  - `/start <username>` привязывает MAX chat id к `MsUser.max_id`;
  - `/unbind` очищает `MsUser.max_id`;
  - `/start` отправляет подсказку.
- Обработан callback `application_done:<id>` через `ApplicationService.transition(..., target_status="done")`.
- Добавлены unit-тесты webhook dispatch и inline-кнопок.

---

## Отклонения от плана

- Бот не регистрировался и webhook не ставился через MAX API: нужен реальный token владельца и публичный prod/staging URL.
- Использовано существующее поле `MsUser.max_id`, а не новое `max_chat_id`.
- `accept_departure:<id>` не реализован: в текущей доменной модели нет статуса/сервиса "accept departure"; есть только create/complete/archive/delete flow.
- Signature реализована через shared secret header, потому точный MAX signature contract в карточке не закреплён.

---

## Архитектурные решения

- MAX webhook размещён в `apps.integrations.max`, а в API подключён через `apps.interface.api.v1.urls`, чтобы внешний endpoint оставался в `/api/v1/`.
- Binding пишет в `MsUser.max_id`, потому это поле уже есть в текущем compat-состоянии и используется `MaxChannel`.
- Логи webhook не пишут chat id, чтобы не тащить PII в structured logs.
