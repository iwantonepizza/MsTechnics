## Что сделано

- Добавлен app `apps.integrations.gmail_alarms` в `INSTALLED_APPS`.
- Добавлен parser `parse_alarm_email()` для `Faulty Alarm Notification` и `Recovery Notification`.
- Parser вытаскивает `device_id`, `screen_name`, timezone-aware timestamp и все `RC(No:N)` / `Receiving card(No:N)`.
- Добавлена модель `AlarmEvent` с raw audit fields, связями на `Display`, `Cell`, `Panel`, idempotency constraint и resolution fields.
- Добавлено поле `Display.vnnox_device_id` и миграция `directory_displays.0002_display_vnnox_device_id`.
- Добавлен service ingestion:
  - `ingest_alarm_email()`;
  - `process_alarm_record()`;
  - mapping `device_id -> Display`;
  - mapping receiving card number to `Cell` через `row/col`;
  - recovery закрывает последний открытый faulty.
- Добавлена management command `pull_vnnox_alarms`.
- Добавлена admin-регистрация `AlarmEvent`.
- Добавлены parser tests на faulty/recovery/unknown subject.
- Добавлен API `GET /api/v1/displays/{slug}/alarms/?resolved=false&limit=50`.
- OpenAPI schema и `frontend/src/shared/api/schema.d.ts` регенерированы.
- В `DisplayViewPage` добавлена вкладка `VNNOX` в правом rail:
  - показывает открытые алармы;
  - отображает cell/panel/status/raw position;
  - кнопка «Создать заявку» открывает существующую `CreateApplicationModal` с prefill-комментарием.
- Добавлен template `vnnox_alarm_unresolved`.
- Добавлена command `check_unresolved_alarms`:
  - threshold берётся из `VNNOX_ALARM_NOTIFY_THRESHOLD_MINUTES`;
  - уведомляет `monitoring/admin/all` пользователей города;
  - idempotency через `Notification.related_target`.
- Добавлены systemd unit/timer для `pull_vnnox_alarms` и `check_unresolved_alarms`.

## Отклонения от плана

- Gmail command использует существующий `Config/token.pickle` legacy flow и не меняет OAuth flow.
- Реальные 4 письма владельца не использовались: в репозитории их нет, тесты сделаны на синтетических fixture по формату карточки.
- API отдаёт paginated response через global DRF pagination; фронт также оставлен совместимым с plain list на случай локального bypass.

---

## Архитектурные решения

- `gmail_alarms` выделен отдельным Django app внутри `apps.integrations`, чтобы модель имела собственные migrations/admin/commands.
- `AlarmEvent.gmail_message_id` добавлен сверх карточки для idempotency при повторном pull того же письма.
- Recovery не создаёт заявку и не трогает панели, только закрывает `AlarmEvent.Type.FAULTY`, как требует защита от шума.
- UI не создаёт заявку автоматически, а только открывает ручную форму с подготовленным текстом.
- Уведомления по висящим алармам используют существующий notification dispatcher и fallback order.
