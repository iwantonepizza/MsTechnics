## Что сделано

- Убран основной runtime blocker migration graph:
  - legacy concrete models в `main`, `user`, `application`, `departure` заменены на compat shim/re-export;
  - `zip` заменён на proxy/compat shim без дублей concrete models;
  - пустые `monitoring`, `control`, `service` удалены из `INSTALLED_APPS`.
- Исправлены legacy runtime references:
  - `Dockerfile` и `docker-compose.yml` используют `Config.wsgi:application`;
  - legacy импорты из `MsServiceControl.settings` переключены на `Config.settings`.
- Добавлены state-only cleanup/import миграции для legacy и новых apps.
- Сгенерированы дополнительные state-alignment миграции для оставшегося drift между текущими моделями и ранними state-import миграциями.
- Все alignment-миграции оформлены через `SeparateDatabaseAndState(database_operations=[])`, чтобы обновить только Django state.
- Исправлен route warning: `me/change-password` больше без ведущего `/`.

## Отклонения от плана

- `mail` и `main_menu` не трогались глубже текущего минимума: они не ломали duplicate-model blocker, а `main_menu` ещё завязан на legacy history модели.
- DB-backed часть acceptance локально не закрыта: `migrate --plan` и `migrate` не проверить без живого PostgreSQL host из env.

Нет других отклонений.

## Архитектурные решения

- Остаточный drift гасился не через переписывание старых `0001_initial_state_import`, а через новые выравнивающие миграции. Это сохраняет историю и делает change set явным.
- Выравнивающие миграции сделаны state-only. Для этого hotfix-а цель была привести в консистентность migration state, не вносить неподтверждённые schema changes в БД без staging/prod копии.
- `zip` оставлен через compat proxy-модели, потому старые импорт-пути ещё живы в legacy-коде и будут окончательно сняты только в `T-5-050`.
