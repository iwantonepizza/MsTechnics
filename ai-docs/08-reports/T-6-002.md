## Архитектурное решение

- Вместо `pgBackRest` реализован repo-side operational minimum на `pg_dump`.
- Причина: карточка задачи допускает этот вариант, а для текущего размера БД и до полного server rollout из `T-6-001` это даёт более простой и реалистичный recovery path.
- Off-host sync по умолчанию запрещён без encryption passphrase.

  - `logs/t6_002_restore.log`
  - `logs/t6_002_restore_verify.log`
  - `logs/t6_002_backups/`

Реальный functional cycle ранее был прогнан локально:

1. Поднят source PostgreSQL 16 container.
2. В source DB восстановлен `db_dumps/mstechnics.dump`.
3. Через `backup-db.sh` создан encrypted backup artifact `*.dump.enc` + `*.sha256`.
4. Поднят отдельный target PostgreSQL 16 container.
5. Через `restore-db.sh` выполнен restore encrypted backup в target DB.
6. После restore сверены контрольные counts:
   - `users=7`
   - `applications=10`
   - `displays=8`
   - `panels=2333`

По логам:

- backup создан: `t6_002_20260513_153805.dump.enc`
- checksum verify прошёл
- restore завершился успешно

Дополнительно:

- `restore-db.sh` переведён на `sha256sum -c` вместо GNU-only `--check`, чтобы checksum verify работал и в BusyBox/Alpine-окружениях
