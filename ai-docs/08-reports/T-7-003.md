### Helpers (`apps/core/users/permissions.py`)

- `get_role_names(user)` — учитывает `_prefetched_objects_cache['roles']` (нет лишних SQL), валится в БД, иначе fallback по `LEGACY_PERMISSION_TO_ROLES[user.permission]`.
- `has_role(user, *names)`, `is_admin(user)`, `has_perm(user, perm)`.
- `role_membership_q(*names) → Q` — генерирует ORM-фильтр для `MsUser.objects.filter(...)`, который покрывает и юзеров с прописанными `roles`, и юзеров с пустыми `roles` через legacy `permission` (нужен для BC-friendly работы триггеров уведомлений до прогона backfill на проде).
- `EXTRA_PERMISSION_CHOICES` — `can_edit_zip_counts`, `can_delete_panels`, `can_send_password_reset`.

### Auth / `/me/`

- `apps/interface/api/v1/auth/serializers.py`: JWT-token теперь содержит `permission` (legacy), `roles` (sorted set), `extra_permissions`.
- `apps/interface/api/v1/me/serializers.py`: `MeSerializer.get_roles` через `SerializerMethodField → get_role_names`, плюс `extra_permissions`. Старое поле `permission` оставлено для совместимости фронта.

### DRF permission classes (`shared/permissions.py`)

## Архитектурные решения

1. **Wave подход** (1 → 2 → 3). Сейчас закрыты Wave 1 (модель + helpers + admin + me + JWT) и Wave 2 (переписаны 30+ мест). Wave 3 (`RemoveField(permission)`) — отдельная задача через 2-4 недели стабильной работы.
2. **`role_membership_q` helper.** Notification triggers фильтруют MsUser через ORM, и `has_role(user, ...)` тут не помогает — нужен `Q`. Я завёл `role_membership_q(*names)`, который генерирует `Q(roles__name__in=names) | Q(roles__isnull=True, permission__in=<legacy mapping>)`. Это снимает гонку: даже если backfill ещё не прогнан на каком-то env, юзеры с legacy `permission` получат уведомления.
3. **JWT и `/me/` отдают и `permission`, и `roles`.** Фронт продолжает работать как раньше; миграция на `roles` — отдельный фронт-PR в follow-up. Никакого «всё сломали в один день» нет.
4. **`extra_permissions` — JSON, не Django Permission.** Test `_grant_zip_edit_permission` использовал `Permission.objects.get_or_create + user.user_permissions.add` — это другая ось (Django built-in). Поправил тест под наш JSON-API; helper в `permissions.py` (`has_perm`) проверяет именно JSON-список.
