## Отклонения от плана

- **Что:** `can_edit_zip_counts` реализован как точечный Django permission-check в storage API без раскатки полного fine-grained ACL.
- **Почему:** полная permission-модель вынесена в blocked-задачу `T-7-003`; для `T-7-005` нужен был только узкий override на запись.
- **Нужно ли обновить архитектурный документ:** нет.

- **Что:** legacy query-параметр `display` перестал влиять на storage queryset.
- **Почему:** storage-модели не связаны с `Display`; старый фильтр обращался к несуществующему полю и был потенциальным источником `FieldError`.
- **Нужно ли обновить архитектурный документ:** нет.

- **Что:** генерация `api-schema.yaml` дала широкий diff вне storage-раздела.
- **Почему:** текущий state `drf-spectacular` в проекте уже нестабилен и перестраивает часть snapshot-а за пределами этой задачи.
- **Нужно ли обновить архитектурный документ:** нет.

## Архитектурные решения

- Кастомный permission оставлен на уровне storage API, а не вынесен в общий ACL-слой, чтобы не заходить в скоуп `T-7-003`.
- `low_stock_threshold` добавлен в существующий `StorageItemMixin`, потому что это минимальное изменение без BC-breaking консолидации storage-моделей.
