## Что сделано

- Реализован backend-сценарий снятия панели с ячейки через `PanelMover.remove_from_cell(...)` с optional `new_condition` и optional `application`.
- Добавлен endpoint `POST /api/v1/panels/{id}/remove/` с payload `new_condition`, `comment`, `application_id`.
- В `PanelSerializer` добавлено поле `active_application_id`, чтобы UI мог показывать warning при ручном снятии без закрытия активной заявки.
- На frontend добавлен mutation `useRemovePanel()` и modal `PanelRemovalModal` с двумя режимами:
  - из контекста заявки: `new_condition` обязателен;
  - ручное снятие: состояние можно оставить без изменений.
- На display page добавлены две точки входа в модалку снятия панели:
  - из карточки выбранной заявки;
  - из карточки панели в выбранной ячейке.
- Добавлены backend-тесты `apps/interface/tests/test_panel_removal.py`.
- Добавлены frontend snapshot/smoke-тесты `frontend/src/features/panels/PanelRemovalModal.test.tsx`.
- Обновлены `api-schema.yaml` и `frontend/src/shared/api/schema.d.ts`.

## Отклонения от плана

- **Что:** в `PanelRemovalModal` добавлен `Dialog.Description` через общий `Modal`.
- **Почему:** это убирает warning Radix в тестах и делает диалог корректнее с точки зрения accessibility.
- **Нужно ли обновить архитектурный документ:** нет.

- **Что:** `ruff` на touched Python-файлах потребовал file-level suppression `RUF001/RUF002` в файлах с русскими строками.
- **Почему:** правило ambiguous-unicode конфликтует с текущей локализацией строк в проекте.
- **Нужно ли обновить архитектурный документ:** нет.

## Архитектурные решения

- Обязательность `new_condition` оставлена на уровне UI, а не serializer/service: backend поддерживает оба сценария снятия панели.
- Привязка к заявке проверяется явно через `application_id`, а `ActivityLog` фиксирует `via_application_id` только когда контекст заявки действительно передан.
