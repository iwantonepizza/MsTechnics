## T-7-008. ConfirmDialog — универсальный диалог подтверждения

### Что сделано

- Новый компонент `frontend/src/shared/ui/ConfirmDialog.tsx`:
  - API декларативный: `<ConfirmDialog open onClose onConfirm title description confirmText cancelText variant loading />`.
  - Default `title = 'Точно?'` — самый короткий confirm-prompt из ответов владельца (Md5).
  - Default `variant = 'danger'` — задача про опасные действия (Архив, Удалить, Невозможно).
  - Внутренний `loading`-state на время выполнения `onConfirm`, чтобы предотвратить double-submit. Внешний `loading` (например, `isPending` от useMutation) тоже учитывается.
  - `autoFocus` на кнопке подтверждения для клавиатурного flow.
  - Использует существующий `<Modal>` (Radix Dialog) и `<Button>` — никаких новых зависимостей.
- Дополнительный хук `useConfirmDialog()` — мини state-machine для бойлерплейта `{open, ask, close, props}`.
- Экспорт через `frontend/src/shared/ui/index.ts`.
- 6 vitest-тестов (`ConfirmDialog.test.tsx`):
  1. Рендер заголовка + описания.
  2. Default title = `'Точно?'`.
  3. `onConfirm` + `onClose` вызываются при подтверждении.
  4. Cancel вызывает `onClose`, **не** вызывает `onConfirm`.
  5. Внешний `loading` дизейблит кнопки.
  6. `useConfirmDialog().ask()` открывает диалог.

### Как использовать в местах опасных действий (Архив / Удалить / Невозможно)

```tsx
import { ConfirmDialog, useConfirmDialog } from '@/shared/ui'

const archive = useConfirmDialog()
const mutation = useMutation(/* archive application */)

return (
  <>
    <Button variant="danger" onClick={archive.ask}>В архив</Button>
    <ConfirmDialog
      {...archive.props}
      onConfirm={() => mutation.mutateAsync(applicationId)}
      title="Архивировать заявку?"
      description="Заявка больше не появится в основной выдаче."
      confirmText="В архив"
      variant="danger"
      loading={mutation.isPending}
    />
  </>
)
```

**Подключение** в конкретные места (TransitionModal/Archive/Delete/Невозможно) — это отдельные точечные правки existing-компонентов, не входят в этот PR. Они будут сделаны при следующих изменениях этих компонентов.

### Что НЕ сделано (осознанно)

- **Императивный API** `await confirm({...})` через провайдер — не реализован. Декларативный достаточен и проще. Если будет 20+ мест с однотипным confirm, можно подумать о провайдере follow-up'ом.
- **Конкретные подключения** в TransitionModal/Delete/Archive — отдельная задача (нужна сверка со списком от владельца, какие именно действия требуют confirm).

## T-7-014. История действий юзера в Profile

### Что сделано

**Backend** — endpoint уже существует: `/api/v1/activity-log/?actor=<username>` (см. `apps/interface/api/v1/activity/views.py:22-23`). Backend-изменений не потребовалось.

**Frontend:**

- `frontend/src/entities/activity/hooks.ts`:
  - Добавлен экспортируемый интерфейс `ActivityLogEntry` (id, event_type, target_kind/id/summary, actor_name, occurred_at, description, comment, payload).
  - Существующий `useActivityLog` типизирован (был `any`).
  - **Новый хук `useMyActivity(username, limit=50)`** — фильтрует по `actor`, `enabled` только когда `username` задан.

- `frontend/src/pages/profile/ProfilePage.tsx`:
  - Импорт `useMyActivity` + `Activity` иконка.
  - Под существующими секциями «Личный кабинет» и «Тема интерфейса» добавлена **третья секция «История действий»**:
    - Spinner на время загрузки.
    - Empty-state «История пуста — ваших действий пока не зафиксировано.»
    - `<ol>` с `<ActivityRow>` для каждой записи.
  - Компонент `ActivityRow` показывает:
    - Локализованный event label (через таблицу `EVENT_LABELS`) → fallback на raw `event_type` если не сматчилось.
    - Target type + id (`· Заявка #42`, `· Панель #7`) — через `TARGET_KIND_LABELS`.
    - Дата и время в формате `dd.mm.yy hh:mm` через `toLocaleString('ru', {...})`.
    - Description **или** comment как сабтекст.

### Локализация event-типов

Поддержаны event_type'ы из существующих сервисов:
- `application.created / transition / archived / deleted`
- `panel.condition_changed / moved / removed / installed`
- `departure.created / completed`

Если бэкенд пришлёт неизвестный event_type — UI покажет raw-имя (мягкий fallback). При появлении новых event-типов в сервисах достаточно добавить запись в `EVENT_LABELS`/`TARGET_KIND_LABELS` без изменения логики.

### Тесты

`ProfilePage.test.tsx` — 3 теста:
1. Empty-state при пустом ответе бэкенда.
2. Рендер 2 записей с локализованным event label и target.
3. Хук дёргает `/activity-log/` с правильными params (`actor: 'ivanov', limit: 50`).
