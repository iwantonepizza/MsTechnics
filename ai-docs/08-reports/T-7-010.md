## Отклонения от плана

- **Что:** маршруты из постановки (`/displays/{slug}`, `/applications/{id}`, `/departures/{id}`) не реализованы в текущей SPA.
- **Почему:** фактическая навигация проекта уже построена иначе; поиск привязан к существующим экранам и query/hash deep-links, чтобы не вводить новые роуты без архитектурной задачи.
- **Нужно ли обновить архитектурный документ:** нет.

- **Что:** frontend search использует локальные типы в [frontend/src/features/search/types.ts](/abs/path/C:/Users/miser/mstechnics/mstechnics/frontend/src/features/search/types.ts).
- **Почему:** после регенерации `api-schema.yaml` search-компоненты попали в YAML, но в текущем `schema.d.ts` они не появились как удобные алиасы для существующего слоя `shared/api/types.ts`; для задачи оставлен локальный typed contract без изменения остального API-слоя.
- **Нужно ли обновить архитектурный документ:** нет.

## Архитектурные решения

- Backend search собран в отдельном модуле [apps/interface/api/v1/search/](/abs/path/C:/Users/miser/mstechnics/mstechnics/apps/interface/api/v1/search), чтобы не размазывать бизнес-логику по существующим viewset-ам.
- Для SQLite/`--nomigrations` тестового режима сохранён fallback без `SIMILARITY()`, а prod-сценарий остаётся на `pg_trgm`.
- SPA-переходы сделаны через реальные точки входа, а не через фиктивные страницы-обёртки, чтобы не плодить второй navigation-path для тех же сущностей.
