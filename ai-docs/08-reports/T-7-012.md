## Что сделано

### `frontend/src/shared/lib/notificationSound.ts` (новый модуль)

- `isNotificationSoundEnabled()` — читает из `localStorage['notificationSound.enabled']`, default `true`.
- `setNotificationSoundEnabled(value)` — пишет `'1'` / `'0'`.
- `playNotificationSound(force = false)` — играет короткий beep (triangle wave 880 Hz, ~150 ms с attack/decay envelope) через **Web Audio API**, без аудиофайла.
- `_resetAudioContextForTests()` — внутренний helper для тестов.

**Почему Web Audio API**, а не `<audio src=mp3>`:
- Не тащим аудиоассет в bundle / `public/`.
- Браузеры блокируют автоплей `<audio>` без user-gesture; у Web Audio API то же ограничение, но мы не падаем — `try/catch` молча гасит ошибку, если `AudioContext` не resumed.
- На практике юзер уже кликал по UI до получения SSE-события (вошёл в систему, открыл страницу), так что ограничение не мешает.

### `frontend/src/shared/lib/sse.ts` (правка)

Добавлен set `SOUND_EVENTS` — белый список event-типов, на которые играть звук. Сейчас в нём **только** `application.create` (A8 от владельца: «контролёр когда мониторщик создал»).

В двух местах handler'а — `es.onmessage` и `addEventListener(eventType, …)` — после инвалидации queries вызывается `playNotificationSound()`. Если event-тип не в `SOUND_EVENTS` — звук не играет.

**Расширять при необходимости** через добавление в `SOUND_EVENTS`. Бизнес-логика звука не размазана по коду — централизована тут.

### `frontend/src/pages/profile/ProfilePage.tsx` (правка)

Новая секция «Звуковые уведомления» между «Тема интерфейса» и «История действий»:
- Иконка `Volume2` / `VolumeX` отражает текущий state.
- Описание: «Короткий сигнал в открытом приложении при появлении новой заявки.»
- Кнопка **«Прослушать»** (`<Button variant="ghost">`) — играет звук через `playNotificationSound(true)` (force, обходит preference); `disabled` когда звук выключен.
- Чекбокс **«Включено / Выключено»** — toggle обновляет localStorage сразу через `setNotificationSoundEnabled`.

## Отклонения от плана

- **Без mp3-файла.** Карточка не настаивала на конкретном source. Web Audio API проще и легче.
- **Звук только на `application.create`.** Расширяем при появлении новых SSE-событий через `SOUND_EVENTS`.
- **Preview-кнопка `force=true`.** Позволяет проверить звук **прямо из Profile**, даже если случайно отключил — это UX-улучшение, в карточке не упоминалось.

---

## Архитектурные решения

- **`SOUND_EVENTS` как Set в sse.ts** — централизация, чтобы не разбрасывать `if (eventType === 'application.create') playSound()` по компонентам.
- **localStorage напрямую**, без отдельного store. Preference per-device, не нужен sync через сервер; React Hook (`useState` инициализированный из localStorage) достаточен.
- **AudioContext lazy + ленивый init.** `_audioCtx` создаётся при первом `playNotificationSound`, не на загрузке модуля — экономим CPU/memory если юзер выключил звук с первого дня.

---

## Что НЕ делаем

- Не делаем mp3/wav-загрузку — overkill для одного beep'а.
- Не делаем разные звуки на разные события — KISS, один beep на критичное (новая заявка).
- Не делаем «не отвлекать звуком в первые N минут» / «звук только в рабочее время» — overengineering. Юзер либо включает, либо выключает.
- Не делаем sync preference через сервер — это per-device настройка.
