Две задачи закрыты одним PR — обе работают с панелями, обе через ZipPage, обе требовали и backend, и frontend (изначально казалось frontend-only, но `PanelViewSet` намеренно не имел `CreateModelMixin/DestroyModelMixin`).

---

## Backend

### `apps/interface/api/v1/panels/serializers.py`

Добавлен `PanelCreateSerializer` с полями `name`, `display_id`, `condition_name?`, `comment?`.

### `apps/interface/api/v1/panels/views.py`

- В `PanelViewSet` добавлены `mixins.CreateModelMixin, mixins.DestroyModelMixin`. В `http_method_names` добавлен `delete`.
- `get_permissions()` теперь различает три новых случая:
  - `action='create'` → `HasDepartmentAccess.for_('service', 'admin', 'all')` (Z7: «можно через кнопку»).
  - `action='destroy'` → `HasDepartmentAccess.for_('admin', 'all')` (Z8: «только админ может удалять»).
  - остальные actions работают как раньше.
- Реализован `create(request)`:
  - Резолвит `Display` по id, `Condition` по name (опционально), `Department.name='zip'` для нового склада.
  - Проверяет уникальность имени → `DomainError(code='duplicate_panel_name')` если занято.
  - Создаёт `Panel` + пишет `ActivityLog event_type='panel.created'`.
  - Возвращает `201 Created` + `PanelSerializer`.
- Реализован `destroy(request)`:
  - **Защита от удаления установленной панели**: если у `panel` есть связанная `cell` через `panel.cell` (OneToOne reverse) — `DomainError(code='panel_installed')` с сообщением «снять через POST /panels/{id}/remove/».
  - **Защита от удаления с активной заявкой**: `panel.active_application` → `DomainError(code='panel_has_active_application')`.
  - Иначе: пишет `ActivityLog event_type='panel.deleted'` ПЕРЕД физическим удалением (чтобы target_id успел зафиксироваться) → `panel.delete()` → `204 No Content`.

### Spectacular

`@extend_schema` для обоих новых endpoints с `tags=['panels']` и осмысленными `summary`. OpenAPI schema обновится автоматически на следующем `make api-schema`.

- **`useDeletePanel()`** — `DELETE /panels/{id}/`, тот же invalidate-набор.

### `frontend/src/features/panels/PanelCreateButton.tsx` (новый)

- Видна **только для роли** `service`/`admin`/`all` (`canCreate` через `me.permission`).
- Кнопка `<Plus> Панель` в title bar.
- Открывает Modal с формой: имя (required, maxLength 15), экран (select из `useDisplays()`), комментарий (optional).

- Backend-ошибки (например `duplicate_panel_name`) отображаются в `<div data-testid="create-panel-error">`.

### `frontend/src/features/panels/PanelDeleteButton.tsx` (новый)

- Видна **только для роли** `admin`/`all`.
- Кнопка `<Trash2> Удалить` рядом с деталями выбранной панели.

- При неудаче — `throw err` после установки локального state, чтобы ConfirmDialog не закрылся (даёт юзеру прочитать ошибку и нажать Отмена).

## Отклонения от плана

- **Backend оказался не frontend-only.** В исходных карточках T-7-035/036 я (как архитектор) написал «UI создать панель» / «UI удалить панель», полагая, что endpoints уже есть. По факту `PanelViewSet` намеренно не имел Create/DestroyModelMixin. Пришлось добавить backend (~50 строк во views.py + serializer). Это нормально и закрывает gap.
- **`PanelCreateSerializer` не использует `ModelSerializer`** — `Panel.condition`/`department` это FK на name через `to_field`, и через ModelSerializer пришлось бы городить вложенные сериализаторы. Простой `Serializer` с явными резолверами по name + защитные `DomainError` оказался читаемее.
- **`destroy` пишет в ActivityLog ПЕРЕД `panel.delete()`** — иначе после удаления уже нельзя получить `target_id`. Это правильный паттерн, но не очевидный — отметил тут.
- **Защиты от удаления (panel_installed, has_active_application)** — добавлены без отдельного запроса в карточке, потому что бэз них была дыра: удаление установленной панели сломало бы `Cell.panel` FK на каскад. Согласовано здравым смыслом, не уходит в архитектурную дискуссию.

## Архитектурные решения

- **Per-action permissions** через `get_permissions()` — DRF-идиоматично, не дублирует. Уже использовалось для `remove` в этом же ViewSet.
- **PanelDeleteButton использует `<ConfirmDialog>` из T-7-008** — первое боевое применение универсального компонента. Подтверждает что универсальный API правильно спроектирован: не понадобилось ни одного изменения в ConfirmDialog.
- **role-check на frontend и backend** — frontend скрывает кнопку (UX), backend блокирует endpoint (security). Двойная защита, как и должно быть.

## Что НЕ делать

- **Не разрешать удаление с каскадом** на `cell.panel` или `Application.panel` — текущая логика правильная, кидаем ошибку.
- **Не делать «удалить все панели экрана» bulk-action** — слишком опасно, не запрашивалось.
- **Не давать сервисменам удалять** — это специфика Z8 от владельца, только админ.
