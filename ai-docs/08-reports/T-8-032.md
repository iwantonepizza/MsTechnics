# T-8-032 — Мониторинг может удалить только свою непринятую заявку

## Что исправлено

- В `frontend/src/pages/display-view/DisplayViewPage.tsx` для `monitoring` в карточке заявки оставлено единственное действие «Удалить» и только в статусе `sent_to_control`.
- Удаление завернуто в `ConfirmDialog`, после подтверждения вызывается `useDeleteApplication()`.
- В `apps/workflow/applications/services.py` автор новой заявки теперь сохраняется по `user.username`, чтобы правило «удалить может только создатель» совпадало с проверкой в API.
- В `shared/permissions.py` создание заявки через API больше не разрешено роли `control`.
- В `tests/test_api_e2e.py` поправлен сценарий `409`: заявку создаёт monitoring, а не control, после чего control проверяет невалидный transition.

## Причина бага

- Во фронте monitoring видел чужие action-paths карточки заявки.
- На бэке свежесозданная заявка сохраняла автора как `full_name`, а проверка удаления шла по `username`, из-за чего monitoring мог потерять право удалить собственную заявку.
- Параллельно `control` всё ещё мог создавать заявки через API, что противоречило round 2 матрице.

## Проверка

- `.venv\Scripts\python.exe -m pytest apps/interface/tests/test_security.py tests/test_api_e2e.py -q`
- `frontend: npm test -- src/pages/display-view/DisplayViewPage.test.tsx`
- `frontend: npm run typecheck`

## Регрессии

- В `apps/interface/tests/test_security.py` добавлены кейсы:
  - `control` получает `403` на `POST /api/v1/applications/`;
  - monitoring может создать заявку на `unrecoverable` панели и сразу удалить её как создатель.
- В `frontend/src/pages/display-view/DisplayViewPage.test.tsx` добавлен delete-flow monitoring через `ConfirmDialog`.
