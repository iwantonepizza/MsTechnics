# T-8-070 — Бесконечный логин при неверных данных

## Что исправлено

- В `frontend/src/shared/api/client.ts` response-interceptor больше не запускает refresh-ветку для самих auth-эндпоинтов:
  - `/auth/login/`
  - `/auth/refresh/`
- Логика refresh/retry сохранена для обычных защищённых запросов, где access token действительно мог истечь.

## Причина бага

- При `401` на `POST /auth/login/` interceptor воспринимал это как обычное истечение access token и пытался сходить в `/auth/refresh/`.
- Если refresh-cookie отсутствовал или тоже был невалиден, `/auth/refresh/` отдавал новый `401`, который снова проходил через тот же interceptor.
- В итоге запрос логина зависал в цикле refresh/retry, а `LoginPage` не получал финальную ошибку для показа toast.

## Проверка

- `npm run typecheck`
- `npm test -- src/shared/api/client.test.ts`

## Регресс-тесты

В `frontend/src/shared/api/client.test.ts` добавлены сценарии:

- `401` на `/auth/login/` не запускает refresh;
- обычный `401` на защищённом запросе всё ещё делает refresh и повторяет запрос;
- `401` на `/auth/refresh/` приводит к logout без рекурсивного refresh-loop.

## Результат

- Неверные логин/пароль теперь сразу пробрасывают ошибку обратно в `LoginPage`.
- У валидных сессий refresh-механизм продолжает работать как раньше.
