# T-8-071 — Каскадное удаление панели админом

## Что сделано

- Вынес удаление панели в сервис `delete_panel()` в `apps/directory/panels/services.py`.
- Перевёл `DELETE /api/v1/panels/<id>/` в `apps/interface/api/v1/panels/views.py` на сервисный сценарий вместо прямого `panel.delete()`.
- Обновил текст подтверждения в `frontend/src/features/panels/PanelDeleteButton.tsx`, чтобы UI не обещал сохранение истории, которая теперь удаляется каскадно.

## Логика удаления

- Если у панели есть активная заявка (`panel.active_application`) — удаление запрещено с `409 panel_has_active_application`.
- Если активной заявки нет — всё выполняется в `transaction.atomic()`:
  - освобождается ячейка (`Cell.panel = None`);
  - удаляются `ActivityLog` по самой панели;
  - удаляются `ActivityLog` по связанным заявкам, иначе после `Application.delete()` оставались бы GenericFK-сироты;
  - удаляются legacy `PanelHistoryReport`;
  - удаляются legacy `ApplicationHistoryReport`;
  - удаляются связанные `Application`, а `ApplicationEvent` уходит каскадом;
  - после фактического удаления панели пишется отдельный audit-event `panel.deleted` без `target`, чтобы сохранить след операции без сиротского GenericFK.

## Нюанс legacy-сигнала

- В проекте жив `application.signals.pre_delete`, который при любом `Application.delete()` заново пишет строку в `history_application`.
- Из-за этого недостаточно удалить legacy-историю до удаления заявок; сервис дочищает `ApplicationHistoryReport` ещё раз после `Application.delete()`.

## Результат

- Админ может удалить панель одной операцией без ручной зачистки связанных архивных сущностей.
- Установленная в ячейке панель теперь корректно удаляется: ячейка освобождается автоматически.
- Панель с активной заявкой по-прежнему защищена от удаления понятной доменной ошибкой.
