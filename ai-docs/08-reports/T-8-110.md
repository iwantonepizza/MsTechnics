## Что сделано

- Исправлена корневая причина пустых ежедневных задач: добавлена forward-only PostgreSQL-миграция
  `daily_task.city_id` с legacy `city.name` на числовой `city.id`.
- DailyTask API ограничен разрешёнными городами; UI различает ошибку загрузки и пустой список.
- На главной добавлены фильтры ленты по периоду 1/2 месяца и сущности действия.
- Подтверждён per-user toggle `show_activity_feed` в Django admin.
- Подтверждено каскадное удаление панели, архивных заявок, ActivityLog и legacy-историй. Добавлены
  проверки admin-only доступа, включая multi-role admin.
- Проведён статический аудит кнопок/select. Исправлены общие light/dark contrast tokens и прямые
  неконтрастные использования; добавлен автоматический WCAG 4.5:1 тест токенов.
- Удалена устаревшая `Wires.Meta.permissions`, из-за которой был migration drift. Актуальное право
  `can_edit_zip_counts` остаётся в `MsUser.extra_permissions`.
- Актуализированы OpenAPI schema/types; `Me.roles` исправлен до `string[]`.
- `AUTH_COOKIE_SECURE` подключён к Django session и CSRF cookies.

## Отклонения от плана

- Повторный вызов удаления `ApplicationHistoryReport` сохранён: legacy post-delete signal создаёт
  финальную запись после удаления заявки.
- PostgreSQL-миграция не прогнана на копии prod: локальный Docker daemon остановлен, локального
  PostgreSQL нет. Это обязательный критерий staging/cutover.
- Ручная визуальная приёмка всех экранов браузером не выполнялась; вместо неё выполнены статический
  аудит и автоматическая проверка contrast tokens.

## Архитектурные решения

- Для ActivityLog использован существующий `kind` prefix filter; контракт API не расширялся новым
  несовместимым фильтром.
- Удаление панели остаётся admin-only и запрещено при активной заявке.
- Миграция daily tasks идемпотентно обрабатывает legacy name/id, умеет продолжить после прерывания
  между drop/rename и останавливается на unmapped данных вместо молчаливой потери задач.
