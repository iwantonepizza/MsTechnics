### T-8-100 — ЗИП: всё ещё максимум 50 панелей в колонке
- **Статус:** 🟢 · **Scope:** FE (+проверить BE-фильтр)
- **Корневая причина (проверено):** пагинация в `usePanels` по `next_cursor` рабочая
  (`shared/pagination.py` отдаёт `next_cursor`), НО в `PanelColumn` (ZipPage) стоит
  `fetchAll: Boolean(displayId)` — догрузка всех страниц включается только когда выбран конкретный экран.
  В общем виде (без выбранного экрана) и в колонке «На экранах» список обрезается на 50.
  Доп. риск: фильтр уходит как `?display=<числовой id>` (`String(model.id)`), а API панелей может ждать иной ключ — проверить, что фильтр реально применяется.
- **Файлы:** `frontend/src/pages/zip/ZipPage.tsx` (`PanelColumn`), `frontend/src/entities/panel/hooks.ts` (`usePanels`), `apps/interface/api/v1/panels/views.py` (фильтр `display`).
- **Что сделать:** включать `fetchAll` для колонок всегда (не только при выбранном экране); сверить, что `?display=` принимает то, что шлёт фронт; для больших колонок (2000+ панелей) добавить виртуализацию/догрузку по скроллу, чтобы не было лагов.
- **Приёмка:** в колонке отображаются ВСЕ панели отдела/экрана, не обрезается на 50; без заметных тормозов.

### T-8-101 — Камера: грузит секунду и закрывается
- **Статус:** 🟢 · **Scope:** FE
- **Корневая причина (проверено):** `DisplayCameraCard` (`DisplayViewPage.tsx`, ~стр.724-738) ставит
  **безусловный** `setTimeout(() => setLoadFailed(true), 5000)` и НЕ имеет `onLoad` на iframe, который бы снимал флаг.
  Поэтому через 5с рабочий iframe ВСЕГДА заменяется фолбэком «Встроить поток не удалось». Это и есть «грузит и закрывается».
- **Файл:** `frontend/src/pages/display-view/DisplayViewPage.tsx` (`DisplayCameraCard`).
- **Что сделать:** убрать безусловный таймаут-провал; добавить `onLoad` на `<iframe>`, который помечает поток рабочим; кнопку «Открыть камеру ↗» показывать **рядом/под** iframe всегда (а не вместо него). Не разрушать iframe по таймеру. Опционально: короткий таймер только чтобы показать кнопку-фолбэк дополнительно, не убирая поток.
- **Приёмка:** камера показывается стабильно и не пропадает; если вендор реально блокирует встраивание (X-Frame-Options) — поток пустой, но кнопка открытия в новой вкладке на месте.

### G1 — БАГ: при неверном логине бесконечная загрузка, нет ошибки

> «при входе с неправильными данными бесконечно грузится и не выдаёт ошибку»

- **ID:** `T-8-070` · **Статус:** ✅ review ([отчёт](08-reports/T-8-070.md)) · **Scope:** FE
- **Файл:** `frontend/src/shared/api/client.ts` (response interceptor, стр.31-57)
- **Причина:** при 401 (неверный пароль) интерсептор пытается `POST /auth/refresh/`. Без валидного
  refresh-cookie он тоже отдаёт 401 → этот же интерсептор снова заходит в refresh-ветку, а `_refreshPromise`
  уже выставлен → ожидание самого себя/повторные попытки → запрос «висит», ошибка не доходит до `LoginPage`.
- **Что сделать:** исключить из refresh-retry логики сами auth-эндпоинты:
  `if (status === 401 && !original?._retry && !original.url?.includes('/auth/login') && !original.url?.includes('/auth/refresh'))`.
  Для `/auth/login/` 401 → сразу `reject(error)`, чтобы `LoginPage.onSubmit` показал toast «Неверный логин или пароль».
- **Приёмка:** неверные данные → мгновенная ошибка в форме, без зависания; верные данные и истёкший токен на защищённых запросах → refresh по-прежнему работает.
