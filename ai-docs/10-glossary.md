# Глоссарий предметной области

Читать **первым**, если ты агент и впервые встречаешься с проектом. Без этого словаря непонятно, чем экран отличается от панели и что такое «ячейка». Код без знания этих слов будет промахом.

---

## Физические сущности

### Экран (Display)

Большой светодиодный экран — конечное устройство, которое видят зрители. Находится в конкретном **городе**. Имеет:

- геометрию: `rows × cols` (ряды × столбцы)
- описание (человекочитаемое имя, напр. «Колизей-большой»)
- slug (для URL)
- ссылку на камеру видеонаблюдения
- электросхему (PDF/JPEG)
- проект (PDF)
- фотографии

Экран состоит из **ячеек**, в каждую из которых вставлена **панель**.

### Панель (Panel)

Отдельный светодиодный модуль. Физический объект, который может быть **перемещён** между отделами:

- `monitor` — стоит в экране (в ячейке)
- `service` — отправлена в сервисный центр на ремонт
- `zip` — лежит в ЗИП (запас), рабочая
- `hand` — у техника на руках

У панели есть **состояние** (condition): `work` / `problem` / `broken` / `unrecoverable` и т.п.
И **статус заявки** (application_status) — когда по панели есть заявка.

Панель уникальна по имени (`name`, например `COLOSSEUM-15`). Имя генерируется автоматически при создании экрана.

### Ячейка (Cell / slot)

Физическое «место» на экране в координатах (row, col). В ячейку вставлена панель. Если панели нет — ячейка пустая.

`position` у ячейки — это порядковый номер слева-направо, сверху-вниз, zero-padded: `01`, `02`, ..., `15`.

**Важно:** `(display, row, col)` уникальны; панель не может быть в двух ячейках одновременно (`UniqueConstraint(fields=['panel'])`).

### Город (City)

Группа экранов. Пользователи имеют `allowed_city` (M2M) — право работать с экранами только в определённых городах.

### Отдел (Department)

Логическая единица, между которыми гоняются панели: `monitor`, `service`, `zip`, `hand`.
**Это НЕ то же, что отдел сотрудника.**
Сотрудник имеет `permission` (`monitoring / control / service / all / admin / technical`).

### Состояние панели (Condition)

Текущее физическое/рабочее состояние: `work`, `problem`, `unrecoverable` и т.д. Определяет, что можно с панелью сделать. Например, `work` → можно ставить в экран, `unrecoverable` → в утиль.

У состояния есть цвет (для UI) и иконка-эмодзи.

### ЗИП (zip)

Склад. Физически находится где-то в офисе. Кроме панелей там лежат:

- **Провода** (Wires) — силовые и сигнальные
- **Хабы** (Hubs) — коммутационные устройства
- **Ламели** (Lamels) — части экрана

У каждого — количество, фото, описание.

## Workflow-сущности

### Заявка (Application)

Основная бизнес-сущность. Возникает, когда панель сломалась.

Жизненный цикл (FSM):

```
[создание заявки мониторингом]
       │
       ▼
application_sent_to_control  (отправлена в контроль)
       │
       ▼
application_apply_in_control (принята контролем)
       │
       ▼
application_sent_to_service  (отправлена в сервис)
       │
       ▼
application_work_in_service  (сервис работает над ней)
       │
       ├──────────────┬──────────────┐
       ▼              ▼              │
      done    application_unable     │
       │              │              │
       ▼              ▼              │
  archive_done   archive_unable      │
                                     │
                          также возможна отмена на этапе
                          application_sent_to_control → удаление
```

**Важно:** в старой архитектуре у заявки 28(!) полей вида `comment_<stage>`, `time_<stage>`, `file_<stage>`, `user_<stage>` — по 4 поля на каждое состояние. В новой архитектуре это нормализуется в отдельную таблицу `ApplicationEvent` / `ActivityLog`.

### История (History / Activity)

Пять разных таблиц, которые логируют события:

- `ApplicationHistoryReport` — события заявок
- `PanelHistoryReport` — события панелей (перемещения, смена состояния, ремонт)
- `DisplayHistoryReport` — события на конкретной ячейке экрана
- `DepartureHistoryReport` — события выездов
- `DailyTaskHistoryReport` — выполнение ежедневных задач

**В новой архитектуре** сливаются в единую таблицу `ActivityLog` с полиморфной связью (через `ContentType` + `GenericForeignKey`) + `event_type`. Это даст экран «все действия в одном месте», которого хочет владелец.

### Выезд (Departure)

Запись о том, что техник поехал на объект. Привязана к исполнителю (Executor). Имеет статус (Создан / выполнено / В архиве).

### Исполнитель (Executor)

**Не то же, что пользователь системы!**
Это техник, который физически выполняет работы. У него есть:

- имя/фамилия
- должность
- телефон
- telegram_id

Пользователь системы (MsUser) — это тот, кто логинится в веб.
Исполнитель — тот, кого назначают на выезд или на заявку.

### Ежедневное задание (DailyTask)

Задача с расписанием. Например, «утренний осмотр экранов города X с 9:00 до 11:00». Движок `daily_checker.py` каждую минуту проверяет их и шлёт уведомления: за 5 минут до старта / при старте / за час до конца / при просроке / по выполнению.

## Интеграционные сущности

### GmailMessage + Alarm

От китайской системы VNNOX приходят письма-отчёты об авариях на экранах. `mail.views.get_emails` парсит их через Gmail API. На каждое письмо создаётся `GmailMessage`, внутри — список `Alarm` (таблицы аварий из HTML письма).

### Telegram (TG)

Канал уведомлений сотрудников. Каждому `MsUser` и `Executor` можно проставить `telegram_id`. Сервис `sender_tg_message.py` слушает Redis и шлёт сообщения через Bot API.

**Проблема:** в РФ с недавних пор режется/блокируется трафик к `api.telegram.org`. Решение в `06-integrations/telegram-russia-workaround.md`.

### МАХ

Мессенджер VK Group. Планируется как альтернатива/дубль TG. Клон той же логики: хранить `max_id` у пользователя, адаптер поверх API МАХ, та же очередь.

## Роли и права доступа

| permission | Что может                                                          |
|------------|--------------------------------------------------------------------|
| `admin`    | Всё, включая админку Django                                        |
| `all`      | Работа во всех трёх отделах                                        |
| `monitoring` | Создавать заявки, менять состояние панелей (`work`/`problem`)    |
| `control`    | Принимать заявки, отправлять в сервис, архивировать              |
| `service`    | Выполнять ремонт, фиксировать результат, работать с ЗИП         |
| `technical`  | Внутренний: получает технические уведомления                     |
| `none_type`  | Никаких прав (заглушка)                                          |

Плюс ограничение по `allowed_city` (список городов, в которых юзер имеет доступ).

**В новой архитектуре:** `permission` заменяется на полноценные Django Groups + Permissions, а проверки идут через DRF permissions_classes.

## Чего в системе НЕТ (и не должно появиться без ADR)

- **Платёжной системы.** Это внутренний ops-инструмент, не торговля.
- **Клиентских аккаунтов.** Все пользователи — сотрудники.
- **Публичного API.** API SPA — внутреннее, с JWT.
- **Мультиязычности.** Только русский. Верстка, тексты, админка — всё на русском.
- **Многотенантности.** Одна компания, одна база.

Если задача подразумевает что-то из этого списка — она **неправильно поставлена**, иди к архитектору.
