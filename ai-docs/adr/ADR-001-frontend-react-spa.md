# ADR-001. Фронтенд: React SPA + DRF, а не HTMX / Alpine

**Статус:** принято
**Дата:** 2026-04-22
**Автор:** архитектор (Claude Opus) по согласованию с владельцем

---

## Контекст

Проект MsTechnics — внутренний инструмент для трёх отделов (Мониторинг, Контроль, Сервис) + ЗИП. Текущая версия — Django templates + ванильный JS + `open-modal` fetch-HTML-в-overlay. Это работает, но:

- Любое действие = перезагрузка страницы или HTML-фрагмент из сервера, без клиентского кэша
- Модалки собираются на сервере (`.fetch(apiUrl).then(res => modalContent.innerHTML = html)`) — небезопасно, мешает XSS-защите, тяжело тестировать
- Нет реактивности: смена состояния заявки не видна коллегам без перезагрузки
- Плотность работы пользователя (монитор смотрит 8 часов в день) требует keyboard-first UX, мгновенных переходов, URL-state — всё это на SSR-стеке с overlay-fetch не делается нормально

Обсуждались три варианта:

### (a) Оставить Django templates, улучшить UX
- Плюсы: минимум изменений, один стек
- Минусы: невозможно сделать плотный UX без клиентского state — а именно он нужен для работы 8 часов/день

### (b) HTMX + Alpine.js + Tailwind поверх Django
- Плюсы: меньше кода на фронте, бэк единый, можно идти инкрементально
- Минусы:
  - optimistic updates через HTMX неудобны
  - real-time (SSE invalidation by query-key) — руками
  - сложная interactive grid 100 cells с правой панелью и вкладками — это уже граница применимости HTMX
  - нет компонентной библиотеки того же уровня что shadcn/ui

### (c) Отдельная React SPA + DRF — **принято**
- Плюсы:
  - TanStack Query решает server state
  - shadcn/ui + Tailwind — быстрый и качественный UI
  - real-time через SSE + query invalidation — паттерн, не изобретение
  - Storybook + e2e на Playwright
  - фронт и бэк можно разрабатывать параллельно после фиксации API-контракта
- Минусы:
  - два стека (Python+TS), два процесса CI/CD
  - JWT/refresh flow + CORS — есть boilerplate
  - developer на фронте обязан понимать FSD/query-cache

---

## Решение

Принимаем вариант **(c)**. Стек:

- **Backend:** Django 5.1 + DRF 3.15 + SimpleJWT + Redis Streams
- **Frontend:** отдельный пакет `frontend/`, React 18 + TS + Vite + TanStack Query + Zustand + Tailwind + shadcn/ui + React Hook Form + zod
- **Real-time:** SSE (`text/event-stream`) на канал `user:<id>:events`
- **Деплой:** nginx обслуживает статику SPA + проксирует `/api/` на gunicorn

---

## Последствия

### Хорошие
- Тестируемость: бэк — pytest, фронт — Vitest + Playwright, независимы
- Возможность переиспользовать API для мобильного приложения (если появится)
- Чёткая граница бэк/фронт → меньше случайной связанности
- Разработка параллельная: после фиксации API — фронт и бэк работают независимо

### Плохие (митигированы)
- Два стека → **требует** архитектора с кругозором по обоим (есть)
- API-контракт должен быть зафиксирован раньше кода → отсюда `ai-docs/07-frontend/api-contract.md`, ведётся осознанно
- JWT + CORS boilerplate → списано в T-3 phase, стандартные либы

### Миграция
- Legacy Django templates работают параллельно в Фазах 1-2 (чиним баги и делаем модели)
- Фаза 3: DRF API становится каноном, templates перестают принимать мутации, только просмотр
- Фаза 4: SPA полностью заменяет legacy
- Фаза 5: удаляем templates + `*/views.py` (оставляем только admin views)

---

## Альтернативы, которые могут быть рассмотрены повторно

- Если окажется что компания не может содержать отдельного FE-кодера на долгом промежутке — возврат к HTMX обсуждается, но это будет новый ADR и переписывание уже сделанного.
- React Server Components / Next.js — **не рассматриваем сейчас**, проект не SEO-ориентированный, рендер на сервере не даёт выгоды.
